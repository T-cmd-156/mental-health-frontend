## 一、文档说明与使用指南

### 1.1 文档目的

本文档按《移动端数据库说明文档》的结构整理，基于当前前端仓库的类型定义、Mock 持久化与接口字段，给出后端可直接对照的表结构建议。

### 1.2 字段命名约束

- 仅使用前端已存在变量名，不做重命名。
- 当前项目同时存在 `camelCase` 与 `snake_case`（如 `studentId`、`create_time`、`created_at`），后端请保持一致或提供映射。

### 1.3 数据来源

- 类型定义：`src/types/*.ts`
- Mock 持久化：`src/mock/db.ts`
- 业务数据与接口：`src/mock/*.ts`、`src/api/*.js|ts`

## 四、表间关系

### 4.1 主要关系

- `students.user_id` → `users.id`
- `consultants.user_id` → `users.id`
- `teachers.user_id` → `users.id`
- `students.college_id` / `teachers.college_id` / `consultants.secondary_station_college_id` → `colleges.id`
- `appointments.studentId` → `students.id`
- `appointments.counselorId` → `consultants.id`
- `appointment_records.appointment_id` → `appointments.id`
- `schedule_periods.schedule_id` → `schedules.id`
- `schedule_periods.appointment_id` → `appointments.id`
- `cases.studentId` / `crises.studentId` → `students.id`

### 4.2 关系图（简）

```
users ─┬─ students ─┬─ appointments ── appointment_records
       ├─ teachers  │        │
       └─ consultants┘        └─ schedule_periods ── schedules

colleges ─┬─ students
          ├─ teachers
          └─ consultants

students ─┬─ cases
          ├─ crises
          └─ crisis_cases
```

## 六、与后端比对

| 项目 | 前端当前字段 | 后端实现字段 | 处理方式 |
|------|--------------|--------------|----------|
| 命名风格 | `studentId` + `created_at` 混用 |  | 建议后端保持原名或网关映射 |
| 预约时间 | `date` + `create_time` + `end_time` |  | 保持三字段拆分 |
| 危机主模型 | `crisis_cases`（复杂嵌套）+ `cases/crises`（轻量） |  | 可统一为主表+子表结构 |
| 配置数据 | `visit_form_config` `scale_config` `consent_config` |  | 建议落库，避免硬编码 |
| 模板数据 | `week_templates` + `MOCK_SCHEDULE` |  | 建议提供正式排班模板表 |

## 一、文档说明与使用指南

本文档基于前端项目（Mock 数据、数据模型、API 文档）逆向整理，汇总**前端所需的数据结构与字段**，供后端设计与实现数据库时参考。

| ② 比对反馈 | 若后端已有数据库设计，请将差异填写在 [第六章](#六与后端比对) 表格中，并反馈前端 |
